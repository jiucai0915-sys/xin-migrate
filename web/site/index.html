核心差异化 · Verify & Self-Heal　★ 全场重点

模型越弱，编排越值钱

端侧开源模型（Qwen2.5-Coder）比 GPT-4 弱、会犯错。云端套壳谁都会；难的是让弱模型在内网把活干对——这才是护城河。下面是它处理 Oracle (+) 外连接时，真实的「改错 → 读报错 → 自我修复」：

❌ 第一次改写（验证失败）
SELECT
c.cust_id,
COALESCE(c.cust_name, '未知客户') AS cust_name,
CASE o.status WHEN 1 THEN '已支付' ... END AS status_text
FROM customers c, orders o
WHERE c.cust_id = o.cust_id(+)
LIMIT 100;
🔴 残留 Oracle (+) 外连接，目标库不支持
→读懂报错
自我修复
✅ 自修复后（验证通过）
SELECT
c.cust_id,
COALESCE(c.cust_name, '未知客户') AS cust_name,
CASE o.status WHEN 1 THEN '已支付' ... END AS status_text
FROM customers c
LEFT JOIN orders o ON c.cust_id = o.cust_id
LIMIT 100;
🟢 (+) → 标准 LEFT JOIN，静态校验通过 ✓ PASS
💡 让弱模型犯错后还能自己改对，正是「验证 + 自修复」闭环的工程价值。Weak model + strong orchestration.
两道质量关 · Two Quality Gates

从「语法对」做到「结果对」

改完就完事？不。我们设了两道关，能跑通 ≠ 迁移对。

①语法关 Syntax

残留 Oracle 语法检测：NVL、DECODE、(+)外连接、ROWNUM、CONNECT BY、MERGE、ROWID…… 没改干净直接判失败，触发自修复。

②语义关 Semantics

在样本数据上真实执行迁移后的查询、与标准答案逐行比对。专抓"语法对但语义错"——比如把 LEFT JOIN 误写成 INNER JOIN，悄悄丢掉没有订单的客户。

🧪 这种错，语法检查永远查不出，只有真跑数据才抓得到。
